# 飞书妙搭作为宿主平台——Orchestrator 的外部平台认知

> 本文档与 `cowork-as-host-platform.md` 并列，共同构成 orchestrator 的双平台认知层。

---

## 1. 飞书妙搭不是什么

| 错误理解 | 会导致什么 |
|---------|----------|
| "一个接 handoff 的页面工具" | 把 handoff 简化成"页面截图 + 字段列表"，忽略数据对象、流程、权限、连接器等维度 |
| "普通静态页面托管器" | 忽略它具备数据对象建模、流程编排、企业权限集成等能力 |
| "纯前端搭建器" | 以为只有 UI 层需要 handoff，忽略数据层、逻辑层、权限层的交接 |
| "云 IDE / 代码运行环境" | 过度假设它能运行任意代码，而它的核心是低代码配置而非代码执行 |
| "Cowork 的弱化版" | 用"谁更强"的框架理解两者关系，导致错误的任务分配 |

---

## 2. 飞书妙搭更像什么

**飞书妙搭是飞书企业生态内的应用承载与低代码配置平台。**

它的核心特征不是"代码执行能力"，而是：

### 2.1 飞书企业生态内的应用承载

- 应用发布后，企业内用户通过飞书客户端直接访问
- 继承飞书企业的组织架构、权限体系、身份认证
- 不需要自建认证、不需要独立部署、不需要域名

**确定性：高** — 飞书应用平台的公开核心能力。

### 2.2 数据对象与表单建模

- 多维表格 / 数据表作为结构化数据承载层
- 支持字段定义、关联关系、视图配置
- 数据可通过 API / 连接器被外部系统读写

**确定性：高** — 多维表格和数据对象是飞书妙搭的公开核心功能。

### 2.3 页面与组件搭建

- 可视化拖拽搭建页面
- 支持表单、列表、详情、看板等常见数据型页面模式
- 组件绑定数据对象字段

**确定性：高** — 低代码平台的基础能力。

### 2.4 流程与自动化配置

- 工作流 / 审批流配置
- 连接器对接第三方系统
- 定时任务 / 事件触发

**确定性：中高** — 飞书应用引擎公开文档有相关说明，具体能力边界可能因版本和企业配置而异。

### 2.5 企业权限与组织上下文

- 应用级权限控制
- 基于飞书组织架构的角色/部门权限
- 数据行级权限（如适用）

**确定性：中高** — 企业应用平台通常具备，公开文档有权限相关说明。

### 2.6 AI 助手 / 智能组件

- 飞书生态内可能支持 AI 助手接入
- 智能表单、智能搜索等增强组件

**确定性：中** — 飞书在持续增加 AI 能力，但在妙搭中的具体可用形态需要实际验证。

## 3. 飞书妙搭与 Cowork 的关系：不是"谁更强"

### 错误的比较方式

❌ "Cowork AI 能力更强，飞书妙搭页面能力更强"
❌ "复杂任务用 Cowork，简单任务用飞书妙搭"
❌ "Cowork 是开发环境，飞书妙搭是运行环境"

### 正确的比较维度：宿主边界、数据边界、生态边界

| 维度 | Cowork | 飞书妙搭 |
|------|--------|---------|
| **核心角色** | 开发 / 编排宿主平台 | 飞书生态内应用承载与配置平台 |
| **数据边界** | 公司内部数据（MCP）、外部公开数据（WebFetch/WebSearch）、代码库、文件系统 | 飞书体系内数据对象（多维表格）、飞书组织数据、飞书应用上下文 |
| **生态边界** | AI 推理 + 工具链 + MCP 扩展 + skill 体系 + 自动化调度 | 飞书客户端 + 企业权限 + 组织架构 + 连接器 + 飞书应用市场 |
| **典型承载** | 逻辑编排、代码实现、AI 处理、数据分析、review/audit、artifact 产出 | 企业用户可访问的应用页面、数据对象维护、流程配置、权限受控的信息展示 |
| **不足** | 直接承载飞书企业内最终用户的应用访问入口 | 执行复杂代码逻辑、访问非飞书体系的内部数据、多 agent AI 编排 |

### 关键洞察

两个平台的区别**不是能力高低**，而是**它们各自能触达的数据边界和用户边界不同**：

- **Cowork 能触达的**：公司内部系统数据（通过 MCP）、外部公开信息、代码库、AI 推理能力——但它**不能直接承载飞书企业内最终用户的应用访问**
- **飞书妙搭能触达的**：飞书组织内的用户访问入口、企业权限体系、飞书数据对象——但它**不能直接访问非飞书体系的内部数据或运行复杂代码逻辑**

## 4. Cowork 与飞书的协作关系

### 不是"谁更强"，也不是"只有 handoff"

两个平台的区别不是能力高低，而是**它们各自能触达的数据边界和用户边界不同**。同时，Cowork 可以通过 **Local Code Sync** 路径直接修改飞书导出的代码——但这**不等于飞书变成了 Cowork 的子项目**，数据/用户/生态边界不变。

### Orchestrator 的任务分工判断

```
这个任务主要依赖什么宿主边界？
├── 主要依赖：内部数据、代码库、AI 编排、MCP → Cowork 承载
├── 主要依赖：飞书用户访问、企业权限、飞书数据对象
│   ├── 云端配置（页面/数据/权限/流程） → mode: handoff, delivery_mode: cloud_config
│   └── 代码修改（前后端/组件/API） → mode: code, delivery_mode: local_code_sync
└── 两边都涉及 → 拆成多个独立 scope，分别路由（不允许单个 scope 同时包含两种 delivery_mode）
```

---

## 5. Handoff 分类方案（适用于 `delivery_mode: cloud_config`）

> **Handoff = 飞书云端配置类交付物**。`delivery_mode: local_code_sync` 路径下，交付物是修改后的代码文件，不走 handoff。

当 orchestrator 判定某部分任务应由飞书妙搭**云端配置**承载时，handoff 产物应按类型分开：

| Handoff 类型 | 内容 | 示例 |
|-------------|------|------|
| **UI/Page** | 页面结构、区块划分、组件绑定、交互行为 | "新闻展示页需要：列表视图 + 详情抽屉 + 分类 Tab" |
| **Data/Object** | 数据表设计、字段定义、关联关系、视图配置 | "创建 NewsArticle 表：title/summary/category/importance/url/publishedAt" |
| **Workflow/Connector** | 流程配置、触发规则、第三方连接 | "每日 8:00 触发数据刷新，从外部 API 拉取新数据写入表" |
| **Permission/Role** | 权限规则、可见性、数据行权限 | "全员可读，仅管理员可配置信息源" |
| **AI Assistant** | 智能组件或 AI 助手配置（如平台支持） | "在搜索组件中接入 AI 语义搜索" `[NEEDS_VERIFICATION]` |

每个 handoff 类型产出独立的结构化文件，写入 `handoffs/` 目录。

## 6. 飞书妙搭的两个协作入口

### 路径 A：Cloud Config Surface（`delivery_mode: cloud_config`）

通过飞书云端完成配置，交付物是 handoff 文件。

**适合**：
- 页面搭建（拖拽/组件绑定）
- 数据对象建模（多维表格/字段/关联）
- 权限配置（角色/部门/行级）
- 流程配置（审批流/自动化/触发器）
- 连接器（第三方 API 对接）
- 平台内 capability / 插件验证

### 路径 B：Local Exported Code Workspace（`delivery_mode: local_code_sync`）

将飞书妙搭生成的 codebase 下载到本地，由 Cowork/Claude Code 修改代码，用户自行回传飞书云端。

**适合**：
- 前后端代码修改
- 组件/页面/API 逻辑调整
- 类型/样式/业务逻辑实现
- 本地调试与构建

### 路径选择规则

```
if scope 是页面/数据对象/权限/流程/连接器配置:
    → 路径 A（cloud_config）
elif scope 是前后端代码/组件/API 逻辑修改:
    → 路径 B（local_code_sync）
elif scope 两者都涉及:
    → 拆成两个独立 scope，分别走路径 A 和路径 B
```

### 路径 B 的边界约束

| 约束 | 说明 |
|------|------|
| 本地无完整云端数据库 | 数据层需要 mock/stub，不能直接连飞书 DB |
| 本地无组织权限上下文 | 权限相关逻辑无法在本地完整验证 |
| 部分插件不可本地验证 | 飞书平台插件 API 可能本地不可用 |
| **本地通过 ≠ 飞书云端通过** | 回传后仍需在飞书云端调试验证 |

### 路径 B 的数据库约束（P0-3 补充，来源：amhub-insights-v1 复盘）

飞书妙搭云端数据库 **不等于** 外部 DATABASE_URL 直连。设计数据访问方案前必须先判断：

| 场景 | 正确做法 | 错误做法 |
|------|---------|---------|
| 云端表已存在且有数据 | 通过 `@Inject(DRIZZLE_DATABASE)` 直接读取 | 建独立 ETL 脚本用 DATABASE_URL 连 |
| 云端表已存在但为空 | 走 NestJS sync 端点 import（参照现有 sync 模块模式） | 写独立 Node.js 脚本直连 DB |
| 云端表还不存在 | 产出建表 SQL + handoff 给飞书执行，然后走 sync 端点 | 在代码中 auto-migrate |

**核心原则**：数据访问必须通过飞书平台的 DI 注入机制（`@Inject(DRIZZLE_DATABASE)`），不能假设有独立的 DATABASE_URL 环境变量可用。

### 路径 B 的端到端闭环

```
人类操作          DevFlow 职责                     人类操作              可选后续回合
──────────       ──────────────                   ──────────            ─────────────
① 从飞书导出      ② orchestrator 识别项目          ④ 用户回传飞书         ⑥ 如需云端配置，
   codebase          在 workspace 中              ⑤ 用户在飞书云端          回到 Cowork
   到本地        ③ full-stack-developer               调试 + 验证           出 cloud_config
                    修改代码 + 产出                                        handoff
                    impl report
```

- **下载和回传是纯人类操作**，DevFlow 不介入
- DevFlow 职责边界：**修改代码 + 产出 implementation report（含 cloud_validation_items）**
- 回传后如需云端配置（数据源绑定、权限设置等），用户可回到 Cowork 让 orchestrator 出具 `cloud_config` handoff

### 飞书妙搭 Fullstack 项目技术栈

| 层 | 技术 |
|----|------|
| 前端 | React 19 + TypeScript + Tailwind CSS 4 + Rspack |
| 后端 | NestJS 10 + Drizzle ORM + PostgreSQL |
| UI 组件 | shadcn/ui + Radix UI |
| 状态管理 | Zustand / TanStack Query |
| 构建 | Rspack（替代 Webpack） |
| 运行环境 | Node.js >= 22, npm >= 10 |

**项目结构**：
```
├── client/                    # 前端代码
│   ├── src/pages/            # 页面组件（PascalCase）
│   ├── src/components/ui/    # shadcn/ui 组件
│   ├── src/components/business-ui/ # 业务组件（UserSelect, UserDisplay 等）
│   ├── src/api/              # API 请求封装
│   └── src/hooks/            # React Hooks
├── server/                    # 后端代码（NestJS 模块）
│   ├── modules/              # 业务模块
│   ├── database/             # Drizzle Schema
│   └── capabilities/         # 插件配置文件
├── shared/                    # 前后端共享类型
│   └── api.interface.ts
```

**本地 vs 云端差异**：

| 功能 | 本地 | 云端 |
|------|------|------|
| 数据库 | 需配置连接或 mock | 自动连接 |
| 插件调用 | 需要配置凭证 | 自动鉴权 |
| 用户认证 | 需模拟用户 | 自动处理 |
| 构建部署 | 本地构建 | 云端构建 |

**上传方式**：
- 方式一：直接覆盖（少量文件修改）
- 方式二：Git 同步（大量文件修改，如云端支持 Git）

### 已验证前提 vs 待验证前提

| 前提 | 状态 | 来源 |
|------|------|------|
| 飞书妙搭 codebase 可以导出到本地 | ✅ 已确认 | 用户确认 |
| 本地修改后可以上传回飞书云端 | ✅ 已确认 | 用户确认 |
| 本地不拥有完整云端数据库 | ✅ 已确认 | 用户确认 + 飞书官方文档 |
| 飞书导出项目的具体结构 | ✅ 已确认 | 飞书官方文档：React 19 + NestJS 10 Fullstack，`client/` + `server/` + `shared/` |
| 上传回传的具体方式 | ✅ 已确认 | 飞书官方文档：直接覆盖文件 或 Git 同步 |
| 本地修改上传后是否会被飞书重新生成覆盖 | `[NEEDS_VERIFICATION]` | 飞书文档未明确说明，首次实际操作后确认 |

## 7. 不确定性声明

| 能力 | 确定性 | 说明 |
|------|--------|------|
| 页面搭建 + 数据对象 | ✅ 高 | 低代码平台核心能力 |
| 企业权限集成 | ✅ 高 | 飞书应用平台标配 |
| 流程/连接器 | ⚠️ 中高 | 公开文档有提及，能力边界因版本而异 |
| AI 助手/智能组件 | ⚠️ 中 | 飞书在扩展 AI 能力，妙搭中的具体形态需验证 |
| 外部 API 调用能力 | ⚠️ 中 | 连接器可能支持，调用方式和限制需验证 |
| 数据行级权限 | ⚠️ 中 | 可能依赖版本和配置 |

Orchestrator 在生成 handoff 时，如果涉及不确定能力，应标注 `[NEEDS_VERIFICATION]`。
