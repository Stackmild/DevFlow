# Web App Architect Skill

## A. Skill 使命

本 Skill 负责把**产品需求、交互方案、页面设计**转化为一套**可开发、可扩展、可维护**的 Web App 架构。

它不直接产出 PRD、视觉稿或详细表结构，而是回答这些更底层的问题：

- 这个产品应该拆成哪些模块？
- 页面、组件、服务、数据之间的边界在哪里？
- 应该用什么渲染与数据获取方式？
- 哪些状态属于页面本地，哪些属于共享状态，哪些本质上是 server state？
- API、数据模型、目录结构、权限边界应该如何统一？
- 现在先做什么，哪些抽象应该暂缓，未来如何演进？

---

## B. 与 `backend-data-api` 的边界

这份 Skill 负责的是**高层蓝图**，不是详细后端规格书。

### 本 Skill 负责
- 应用拓扑：modular monolith / 单体 + worker / BFF 等
- 模块边界与依赖关系
- 路由树与页面模式
- 渲染/数据获取策略
- 状态边界
- 目录结构与代码层次
- 关键架构 ADR
- 分阶段演进路线

### `backend-data-api` 负责
- 领域实体与表设计
- API 资源/动作边界
- 请求/响应/错误结构
- 列表查询协议
- 写操作幂等与并发规则
- 真相层级（raw / extracted / review / canonical / derived）
- migration / backfill / expand-contract

### 交接规则
- 本 Skill 可以提出**候选实体、候选 API 面**，但不直接细化到最终 schema。
- `backend-data-api` 默认在本 Skill 给出的**模块地图、页面模式、权限边界**之内细化设计。
- 如果 `backend-data-api` 发现模块边界本身有问题，应明确回抛，而不是静默改写整体拓扑。

## C. 适用场景

### 适合
- 从 0 到 1 设计一个 Web App 的整体结构
- 需求已大致明确，但担心后续越做越乱
- 已有若干页面或功能，需要做一次高层架构收口
- 要决定 **modular monolith / 微服务 / BFF / SSR / SPA / hybrid** 等架构方向
- 要为 AI 协作开发提供"先定边界、再让模型写代码"的骨架
- 内部工具、管理后台、工作流系统、Dashboard、AI-native Web App

### 不适合
- 只需要单个页面的 UI 优化
- 只需要具体代码修 bug
- 只需要数据库 SQL 或接口文档局部修订
- 只需要详细表结构、索引、幂等策略或 migration 方案（应交给 `backend-data-api`）
- 只需要发布流程、CI/CD、监控、性能专项优化（应交给专门 Skill）

---

## D. 核心目标

本 Skill 的目标不是"设计一个看起来高级的架构"，而是：

1. **让结构为业务服务，而不是为了炫技而复杂化**
2. **让一个小团队/单人也能长期维护**
3. **让 AI 生成代码时有清晰边界，不会把逻辑写散**
4. **让未来的扩展发生在模块之间，而不是在页面里缠绕**
5. **让关键决定被记录下来，未来能解释为什么这样做**

## E. 默认立场（Strong Defaults）

除非另有明确说明，优先采用以下默认立场：

| # | 默认立场 | 核心规则 | 例外/注意 |
|---|---------|---------|---------|
| E1 | 优先 framework | React 体系优先带约定的框架；不推荐 CRA | 现成框架明显不适配时才自拼 |
| E2 | modular monolith | 单一部署单元 + 明确模块边界 + 层次分离 + 保留未来拆分可能 | 小团队/单 Web 客户端/业务边界仍变化时默认选此 |
| E3 | 不做 micro-frontend | 默认单一前端 + 统一路由 + 统一组件系统 | 多团队并行独立发布 + 生命周期明显不同 + 强组织边界同时成立时才考虑 |
| E4 | 单 Web 客户端不做独立 BFF | monolith 内部可做聚合/view model 层；底层有清晰 canonical resources | 多客户端 / 授权边界差异大 / 数据聚合方式明显不同 |
| E5 | contract-first | 先定资源/输入输出/错误/分页，API 约定独立于实现可讨论 | — |
| E6 | 显式区分三类状态 | UI state / Shared client state / Server state；禁止混成大 global store | — |
| E7 | 按路由/页面类型架构 | 沉淀 Dashboard/List/Detail/Edit/Review/Settings 模式 | — |
| E8 | ADR 记录关键决策 | 框架/拓扑/渲染策略/状态方案/不做 BFF/不做微服务的决定写轻量 ADR | — |

## F. 架构原则（Non-Negotiable）

1. **先定边界再谈实现**：先定模块/页面类型/候选实体/API 面/状态边界/权限边界，再进入具体实现。
2. **先解决高频变化点**：优先模块拆分/路由/数据流向/真相来源/共性骨架；不先做"以后也许会有"的扩展点。
3. **减少耦合优先于增加抽象层**：改 A 不破 B；同类页面复用骨架；同类接口遵守同一规则；组件不与业务缠绕。
4. **单一真相来源**：不在前端维护后端可推导数据；canonical truth 由 `backend-data-api` 定义；view model 不是 canonical truth。
5. **渲染策略服务路由特征**：每条路由明确 server/client/static 选择；不把所有页面塞同一种渲染方式；重数据首屏优先服务端/路由级获取。
6. **避免客户端数据获取瀑布流**：优先路由级/并行/服务端预取；不默认"useEffect 再 fetch"作为首屏模式。
7. **状态提升只到必要层级**：能本地 state 不提升；server state 不伪装全局状态；复杂 screen 可用 reducer/context 收口。
8. **目录结构映射架构**：体现路由/业务模块/shared-ui/data-api/domain 层；不随功能堆叠出 utils2/newPages/final_final。
9. **架构支持渐进式演进**：允许模块独立成服务/升级模式/切换数据获取；不要求一次性大重构后才正确。

## G. 工作流程（Skill 执行步骤）

### Step 1. 识别约束与目标
先提炼以下约束；若缺失，则做保守假设并显式标注：
- 用户是谁（内部/外部、多少人、什么角色）
- 产品目标是什么（效率工具、工作流、分析平台、AI 协作平台）
- 主要页面类型有哪些
- 核心实体有哪些
- 数据从哪里来（手填、数据库、第三方 API、文件导入、AI 抽取）
- 权限模型是否只有登录鉴权，还是有角色/可编辑范围差异
- 是否需要实时刷新、离线、后台任务、审批流、导出、文件上传
- 团队规模与技术能力
- 未来 6–12 个月最可能发生的变化是什么

输出：**约束与假设清单**

### Step 2. 选择应用拓扑（Application Topology）
在以下选项中做判断：
- 单体单应用（small monolith）
- **模块化单体（默认）**
- 单体 + 独立后台任务/worker
- 单体 + 少量拆出的高负载服务
- 多服务架构
- 单体 + 独立 BFF

判断时必须回答：
- 我们到底有多少独立团队/独立发布需求？
- 是否真的存在不同客户端形态？
- 哪些边界是业务边界，哪些只是代码量变大？
- 现在拆服务，能带来什么明确收益？
- 增加的复杂度是否值得？

输出：**拓扑选择 + 不选其他方案的理由**

### Step 3. 划分业务模块与边界
按业务能力而不是按页面名称划分模块，例如：
- Auth / Session
- Workspace / Project
- Todo / Task
- Monitoring / Alerts
- Review Queue
- Reports
- Settings / Admin
- AI Assist / Evidence / Suggestions

每个模块都要说明：
- 职责
- 关键实体（候选）
- 对外暴露的能力
- 依赖哪些其他模块
- 哪些共享能力应下沉到平台层，哪些不该共享

输出：**模块地图（Module Map）**

### Step 4. 设计路由与页面模式
先定义页面类型，再映射具体页面：

| 页面模式 | 典型用途 | 固定要素 |
|---|---|---|
| Dashboard | 总览、异常、摘要 | 关键指标、最近变化、快捷入口 |
| List | 查询、筛选、批量操作 | 筛选栏、表格/卡片、分页、批量动作 |
| Detail | 单实体查看 | 标题区、状态区、分栏信息、关联记录 |
| Edit/Create | 数据录入与修改 | 表单区、校验、草稿/提交、错误提示 |
| Review | 人工审核 / AI 结果确认 | 原始证据、结构化结果、差异、高亮、操作栏 |
| Settings | 配置与规则管理 | 配置分组、说明、权限、保存反馈 |

输出：**路由树 + 页面模式映射表**

### Step 5. 定义数据/API 骨架（不是最终 schema）
对核心实体给出：
- 实体名与业务含义
- 主键/外键是否存在
- 生命周期状态是否存在
- 页面会用到哪些关键读取面
- 是否存在导入 / 审核 / publish 链路

API 骨架层面必须输出：
- Resource / action 的初步边界
- 哪些页面需要聚合 view model
- 哪些能力适合独立 query endpoint
- 哪些能力必须交给 `backend-data-api` 进一步细化成正式契约

输出：**候选实体表 + API 面草案**

### Step 6. 定义渲染与数据获取策略
对每类页面明确：
- server fetch / route loader / client fetch 谁负责
- 是否预取
- 是否并行请求
- 是否需要 streaming
- 是否需要 skeleton / partial loading
- 是否允许 stale-while-revalidate
- 哪些数据必须实时新鲜，哪些可以接受缓存

默认规则：
- **重数据、首屏关键、可服务端获取的数据，优先服务端或路由级获取**
- 客户端组件里的数据获取主要用于：
  - 用户驱动的后续查询
  - 高频交互后的局部刷新
  - 仅客户端上下文才可得的数据
- 不把"首屏核心数据 useEffect 拉取"当作默认模式

输出：**按路由的渲染/数据获取矩阵**

### Step 7. 定义状态边界
对每类状态分类：

| 状态类型 | 例子 | 应放位置 |
|---|---|---|
| Local UI state | 弹窗开关、hover、当前 tab | 组件本地 |
| Screen state | 多筛选器、表单草稿、复杂交互流程 | 页面级 / reducer |
| Shared client state | 当前 workspace、跨页面轻量会话态 | 受控共享状态 |
| Server state | 列表数据、详情数据、统计数据 | 查询层 / 数据缓存层 |

必须显式指出：
- 哪些状态不能重复存
- 哪些状态是 derived state，不应单独落库或入 store
- 哪些变化走 optimistic update，哪些必须以服务端返回为准

输出：**状态归属表**

### Step 8. 设计目录结构与代码边界
目录结构应至少体现：
- app/routes 层
- features/modules 层
- shared/ui 层
- shared/lib 层
- data/api 层
- domain / service / model 层（按需要）

如果是 React / Next / Router 体系，应明确：
- 路由目录与业务模块如何配合
- server-only 代码与 client 代码如何隔离
- 页面级 loader / action / mutation 放在哪里
- 哪些组件是 shared ui，哪些只属于 feature

输出：**推荐目录结构 + 命名规范**

### Step 9. 记录关键决策（ADR）
至少记录以下 ADR：
1. 为什么选当前框架/路由体系
2. 为什么选当前应用拓扑
3. 为什么采用当前渲染策略
4. 为什么采用当前状态管理方案
5. 为什么某些能力不提前拆服务 / 不做 BFF / 不做 micro-frontend

ADR 必须包含：
- Context
- Decision
- Consequences
- Rejected alternatives
- Review trigger（什么情况下重新评估）

输出：**ADR 列表**

### Step 10. 给出 phased architecture plan
不要只给理想终局，要给分阶段建议：
- **Phase 1**：最小但不乱的骨架
- **Phase 2**：随着模块增加需要抽象的点
- **Phase 3**：用户量/复杂度上来后才值得引入的能力

输出：**分阶段演进路线图**

## H. 关键判断规则（Decision Heuristics）

### H1. 何时优先 modular monolith
优先 modular monolith，当：
- 只有一个主要 Web 客户端
- 团队很小
- 业务边界还在变化
- 部署与排障成本要尽量低
- 数据模型之间高度相关
- 多数页面共享组件和交互模式

### H2. 何时才考虑拆服务
只有在以下条件明确成立时才建议：
- 某能力有显著独立扩缩容需求
- 某能力有独立发布节奏
- 某能力有明确的数据或权限边界
- 某能力已成为主应用的复杂度瓶颈
- 拆出来后整体复杂度净下降，而非上升

### H3. 何时考虑独立 BFF
考虑独立 BFF，当：
- 前端不止一个客户端类型
- 同一后端 API 已被不同前端不断"定制化污染"
- 需要为某一前端做专门的数据聚合与权限裁剪

否则默认不做。

### H4. 何时优先服务端/路由级获取
优先服务端/路由级获取，当：
- 首屏必须立即看到数据
- 数据可由服务端安全获得
- 页面是重数据页
- 可以并行预取多个关键请求
- 需要减少客户端 waterfall

### H5. 何时允许客户端获取
客户端获取更合适，当：
- 查询明显由用户交互触发
- 数据依赖浏览器环境
- 局部刷新比整页刷新更合理
- 需要长驻前端缓存与后台重取

### H6. 何时使用共享状态管理
只有当至少满足其一时才引入共享状态方案：
- 多层级组件传值已明显影响可维护性
- 多个页面需要共享同一轻量会话态
- 某个复杂 screen 需要 reducer 风格的集中状态变更

不要因为"项目都会有 store"就引入全局 store。

### H7. 何时引入后台任务 / 队列 / worker
考虑后台任务或异步 worker，当：
- 文件导入解析较慢
- AI 推理、批处理、通知发送不适合同步阻塞请求
- 某些任务需要重试 / 补偿 / 进度追踪

否则先保持同步链路简单。

## J. 输出风格要求

输出时必须做到：
- **优先给结构，不先给技术名词堆砌**
- **明确事实 / 假设 / 建议**
- **每个建议都说明 trade-off**
- **优先给适合当前团队规模的方案**
- **避免"为了未来可能性"过早复杂化**
- **可直接拿给开发/AI 实施**

如果用户不是程序员：
- 用通俗语言解释每个架构决定
- 少用抽象术语，必须解释术语含义
- 不要默认用户理解 SSR、RSC、BFF、hydration、bounded context 等概念

---

## K. 禁止事项（Hard Don'ts）

### 不要做这些：
- 为了"显得专业"默认推荐微服务
- 在单一内部工具里默认引入 micro-frontend
- 把 route 写成页面内部 `useState` 切换
- 默认用 `useEffect` 作为首屏核心数据获取机制
- 把 server state、表单 state、UI state 全塞进一个 global store
- 在没有明确边界时先引入事件总线、队列、大量中间层
- 把"页面越来越多"误判成"必须拆服务"
- 不记录架构决策理由，导致之后只能靠猜

---

## L. 架构味道检查（Architecture Smell Check）

当出现以下情况时，应主动报警：
- 同一实体在多个模块中定义不一致
- 页面逻辑高度依赖互相 import
- 列表页、详情页、编辑页各自发明一套结构
- API 命名混乱，资源与动作混杂
- 前端为了适配后端缺陷维护大量衍生状态
- 一个 store 既放 UI 状态又放远程数据又放表单草稿
- 新增一个页面时需要复制多个旧页面再"魔改"
- 关键决策只有代码里有，没有文字记录
- 技术栈选型与团队能力明显失配

---

## M. 推荐回答模板

当用户请求 Web App 架构设计时，优先按以下结构回答：

1. **一句话判断**：这更像什么类型的 Web App
2. **推荐拓扑**：为什么是 modular monolith / 其他
3. **模块划分**
4. **路由与页面模式**
5. **数据/API 骨架**
6. **渲染与数据获取策略**
7. **状态管理边界**
8. **目录结构建议**
9. **哪些点交给 `backend-data-api` 继续细化**
10. **当前不建议做的事**
11. **Phase 1 先落地什么**

> 先把边界、页面骨架、数据/API 面、状态归属和决策记录定清楚，再让团队或 AI 去写代码；默认选择最简单但不混乱、最克制但可演进的 Web App 架构。
