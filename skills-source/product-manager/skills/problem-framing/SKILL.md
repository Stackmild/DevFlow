# Problem Framing

本 skill 将用户给出的**功能想法或 solution 表述**改写为**可陈述的问题**，并产出：问题定义、目标用户（谁在什么情境下）、现有替代方案、痛点强度、不做会怎样、为什么现在值得做。当输入质量差或证据不足时，优先在输出中暴露「不确定」与缺失项，不替用户编造清晰需求。

---

## Best for

- 用户只说「想做 XX 功能」或「加个 XX」，尚未说明要解决什么问题
- 需求讨论中 solution 很多、问题定义模糊
- 需要在一页内说清「我们在解决什么、为谁、为何现在做」

---

## Typical triggers

- 用户说「写个 PRD」但只给了功能名或一句话想法
- 用户说「做个 AI 搜索 / 推荐 / 看板」等具体形态，未说问题
- 主 skill 路由到 problem-framing（因问题定义不清）

---

## Not for / When NOT to use

- 问题已经清晰、且用户已明确写出「要解决的问题」与「目标用户」时，不必再做 problem-framing，可交给 outcome-definition 或 prd-writing。
- 输入是一堆未整理的访谈/反馈/研究时，应先做 discovery-synthesis，再视需要做 problem-framing。

---

## Required inputs

- 至少包含：用户或利益相关方提出的**功能想法、需求描述或痛点描述**（可为一句话、一段话或零散要点）。若连「想做什么」都没有，无法执行本 skill，应先向用户索取或澄清。

---

## Optional inputs

- 业务背景、已有数据或调研结论
- 已知的目标用户或场景

---

## Default assumptions when context is missing

- 用户给的往往是 **solution**（例如「做一个 AI 搜索框」），不是 **problem**（例如「用户无法在现有内容中快速找到高相关结果」）。默认需要从 solution 反推或追问 problem。
- 若用户未区分「想要的东西」与「真正遇到的阻碍」，输出中应显式区分并标出「推断」与「不确定」。
- 证据不足时，不自动脑补细节；在输出中用「不确定」「待验证」标注。

---

## Core workflow

1. **提取与区分**：从输入中提取 (1) 用户**想要的东西**（功能、形态）、(2) 用户**真正遇到的阻碍或痛点**（若已提到）。若只有 (1) 没有 (2)，标出「问题未陈述，需澄清」。
2. **问题陈述**：若有足够线索，写出一句可验证的「当前问题陈述」；否则写出「当前能推断的问题方向」并标「待验证」。
3. **目标用户与情境**：谁在什么情境下遇到上述问题？若输入未提供，写出「待澄清」并列出 1～2 个澄清问题。
4. **现有替代方案**：用户今天如何应对该问题（包括不用产品、用竞品、用土办法）？若未知，写「待调研」。
5. **痛点强度与不做后果**：该问题发生的频率、影响范围、若不解决会怎样？能写则写，不能写则标「待验证」。
6. **为何现在做**：为什么值得在当下投入（时机、战略、数据支持等）；若缺失则标「待补充」。
7. **标注证据与假设**：在输出中明确哪些来自输入（Evidence），哪些是推断（Inference），哪些是假设（Assumption），哪些尚不确定。

---

## Output contract

输出默认采用以下结构；可在此基础上增加本 skill 专属段落。

| 层 | 内容要求 |
|----|----------|
| **Task framing** | 本输出是问题框定，不是方案设计；边界为「问题与用户」，不包含 PRD 或功能列表。 |
| **Evidence** | 从输入中直接得到的事实、引用（用户原话或已有结论）。 |
| **Assumptions** | 当前采用的、尚未验证的假设（若存在）。 |
| **Inference** | 从输入材料推导出的中间判断（如从「更快找内容」推导出的问题方向）；须显式标注为 Inference，不得混同为 Evidence。Analysis 内应区分 Evidence、Inference、Assumption、Recommendation 的边界。 |
| **Analysis** | 本 skill 核心产出：见下「必含项」。 |
| **Recommendation** | 明确建议，不超过 3 条（例如：建议先与用户确认 X；建议补足 Y 后再做 outcome-definition）。 |
| **Open questions / Next step** | 还缺什么、建议下一步做什么（如接 outcome-definition 或补充调研）。 |

**必含项（Analysis 内）**：

- **当前问题陈述**：一句或一段可验证的问题描述。
- **谁在什么情境下遇到问题**：目标用户与使用/遇到问题的情境。
- **现有替代方案**：用户当前如何应对（含不解决、竞品、土办法）。
- **痛点强度**：频率、影响、紧急程度（若可知）；否则标「待验证」。
- **如果不做，会发生什么**：不做该事的后果或机会成本。
- **为什么现在值得处理**：时机、战略或数据理由；若缺失则标「待补充」。

---

## Quality bar / Acceptance criteria

- 输出中**没有**把用户给的 solution 直接当作问题定义（例如不能把「做一个 AI 搜索框」当作问题陈述）。
- 若问题无法从输入推导，输出中显式写出「问题未陈述 / 待澄清」及澄清建议，而不是编造一个问题。
- 证据与假设、推断被区分标注。
- 六项必含项均有对应内容或「待澄清/待验证」说明。

---

## Clarifying questions

在以下情况主动向用户澄清：

- 只有功能名（如「智能推荐」）：您希望解决的具体问题是什么？谁会在什么情况下遇到？
- 有痛点但无用户：这些痛点主要发生在哪类用户、什么场景？
- 有用户但无替代方案：用户目前如何应对？有没有在用其他产品或办法？

---

## Common pitfalls

- **Solution 当问题**：把「做一个 XX 功能」直接写成「问题：需要做 XX 功能」，而不是「问题：用户/业务在 Y 情境下遇到 Z 阻碍」。
- **替用户编造**：输入含糊时，编造一个「看起来清晰」的问题陈述，而不是标出「不确定」并列出澄清问题。
- **忽略替代方案**：不写用户今天如何应对，导致无法判断「不做会怎样」和痛点强度。
- **混淆想要与阻碍**：把用户「想要的功能」当成「遇到的阻碍」，未区分需求与问题。

---

## Good output example

**输入**：用户说「我们想做一个 AI 搜索框，让用户更快找到内容」。

**良好输出（节选）**：

- **Task framing**：本输出为问题框定，不包含功能方案或 PRD。
- **Evidence**：用户表述为「AI 搜索框」「让用户更快找到内容」；可推断「找内容」是核心场景。
- **Assumptions**：当前假设「更快」指减少时间或减少步骤；未验证用户是否真的认为「找内容」是痛点。
- **当前问题陈述**：用户/访客在站内**查找与当前意图高相关的内容时**，耗时长或步骤多，导致放弃或满意度低。（若实际是「找不到」而非「慢」，需再澄清。）
- **谁在什么情境下**：使用本站且带有明确信息需求的用户（如找某篇文档、某条配置）；情境待进一步确认（例如是首屏搜索还是结果页筛选）。
- **现有替代方案**：待调研。可能包括：站内现有搜索、手动浏览、用站外搜索再回站。
- **痛点强度**：待验证（需数据或访谈：多少人、多频繁遇到，放弃率或满意度现状）。
- **如果不做**：继续依赖现有搜索或浏览路径；若痛点真实存在，则转化或留存可能受损。
- **为何现在做**：待补充（战略优先级、数据支撑或机会窗口）。
- **Recommendation**：建议先与业务/用户确认「问题主要是慢、还是找不到、还是结果不相关」；再补 1～2 个用户原话或数据点以验证痛点强度。
- **Open questions / Next step**：问题陈述是否与业务理解一致？建议确认后接 outcome-definition，定义「更快/更好找」的可观测标准。

---

## Bad output example

**输入**：同上，「我们想做一个 AI 搜索框，让用户更快找到内容」。

**不良输出（节选）**：

- **问题**：用户需要 AI 搜索框来更快找到内容。
- **目标用户**：所有用户。
- **解决方案**：建设 AI 搜索框。

问题在于：把「做 AI 搜索框」当成了问题本身；目标用户写成了「所有用户」且无情境；没有替代方案、没有痛点强度、没有「不做会怎样」、没有「为何现在做」；没有区分证据与假设，也没有标出任何「待澄清」。这类输出无法支撑后续 outcome 定义与 PRD，且会固化 solution-first 的决策。

---

## Related skills / Handoff rules

- **outcome-definition**：问题框定完成后，通常接 outcome-definition，把「要解决的问题」转为可观测的 outcome 与 success metrics。路径：`skills/outcome-definition/SKILL.md`。
- **prd-writing**：只有在问题与目标用户相对清晰、且已做过 outcome-definition（或至少成功标准已明确）后，才建议进入 prd-writing。若主 skill 直接请求写 PRD 但问题未框定，应建议先完成本 skill 再写 PRD。
- **discovery-synthesis**：若输入是一堆未整理的访谈/反馈，应先走 discovery-synthesis，再视结果做 problem-framing。
