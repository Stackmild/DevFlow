# Experiment Design

本 skill 将**高风险假设**转为**低成本、可执行、可判定成败**的验证方案。从「需要验证什么」出发，产出：验证目标、最小测试方式、样本选择、观测指标、成功/失败阈值、预期成本与时间、风险与限制。强调**最小验证**，不鼓励一上来做大实验；实验必须能**判定**成败，不能只是「先看看反馈」。须能识别虚假验证：样本过偏、指标与假设错位、成本过高、无法区分因果、把「收集反馈」误当验证。与 assumption-mapping 边界清晰：后者识别假设与验证优先级，本 skill 设计具体验证方案。

---

## Best for

- 已有明确「要验证的假设」（通常来自 assumption-mapping），需要可执行的验证方案
- 在投入大规模开发或完整 PRD 前，需要先小范围验证关键前提
- 需要可判定的成功/失败标准，而不是开放式「先上线试试」

---

## Typical triggers

- 用户或主 skill 要求「设计一个验证/实验」
- assumption-mapping 已完成，产出中建议接 experiment-design
- 关键 desirability/feasibility/viability 假设未验证，需在写 PRD 或全量开发前验证

---

## Not for / When NOT to use

- 尚未明确「要验证哪条假设」时，先做 assumption-mapping；本 skill 不负责从零识别假设。
- 仅需做多候选排序或检查已有计划现实性时，用 prioritization 或 roadmap-reality-check。

---

## Required inputs

- 至少包含：**要验证的假设**的清晰陈述（若来自 assumption-mapping 产出更佳）。若没有明确验证对象，无法设计有意义的实验，应先做 assumption-mapping。

---

## Optional inputs

- 假设的上下文（为何重要、若错了会怎样）
- 可用资源（时间、人力、可触达用户、技术能力）
- 已有 baseline 或历史数据

---

## Default assumptions when context is missing

- 验证应从**最小可行**形态开始；不默认做大而全的实验。
- 实验必须能**判定**：有明确的成功/失败阈值，能区分「假设得到支持」与「假设被推翻」；不默认「先看看反馈」即验证。
- 须防范：指标与假设错位、样本选择失真、成本过高、结论不可归因；输出中必须包含风险与限制说明。

---

## Core workflow

1. **明确验证目标**：用一句话写出「本验证要回答什么问题」；必须与输入的假设一一对应。
2. **选择最小测试方式**：在能判定的前提下，选成本最低的方式（如假门、concierge、小流量、单渠道、原型测试）；避免一上来全量或多变量。
3. **样本选择**：谁参与、多少人、如何招募、排除条件；注明可能偏差（如自愿者偏差、渠道偏差）。
4. **观测指标**：与假设直接对应的指标（如假设「用户会配置」→ 指标为配置率/使用率）；区分 leading 与结果指标；避免与假设无关的 vanity 指标。
5. **成功/失败阈值**：在什么结果下判定「假设得到支持」、什么结果下「假设被推翻」；阈值需可观测、可提前设定。
6. **成本与时间**：预期耗时、人力、物料或工具成本；若超过某上限则建议缩小实验范围。
7. **风险与限制**：样本局限、外部干扰、归因困难、无法推广等；明确写清「本实验不能回答什么」。
8. **输出分层**：按 Task framing, Evidence, Assumptions, Inference, Analysis, Recommendation, Open questions / Next step 组织；Analysis 内包含上述各项。

---

## Output contract

输出须采用以下结构，并确保验证可判定、可执行。

| 层 | 内容要求 |
|----|----------|
| **Task framing** | 本输出是单次（或一期）验证方案，不是完整产品方案；边界为「验证某条假设」，不替代 PRD。 |
| **Evidence** | 输入中的假设陈述、已有证据或约束。 |
| **Assumptions** | 本实验设计依赖的次要假设（如「用户愿意参与测试」）。 |
| **Inference** | 从假设推导出的「若验证通过/不通过则意味着什么」；须显式标注。 |
| **Analysis** | 本 skill 核心产出：见下「必含项」。 |
| **Recommendation** | 明确建议，不超过 3 条（如：建议先跑 2 周再决定是否扩量；建议验证通过后接 prd-writing）。 |
| **Open questions / Next step** | 验证后的决策点、建议下一步（如接 prioritization / prd-writing）。 |

**必含项（Analysis 内）**：

- **要验证的假设**：与输入对齐；一句可验证的陈述。
- **验证目标**：本实验要回答的具体问题。
- **最小测试方式**：采用的形式（假门/concierge/小流量/原型等）、触达方式、时长。
- **样本选择**：目标人群、样本量、招募方式、排除条件；样本局限说明。
- **观测指标**：与假设对应的指标、口径、观测时点。
- **成功/失败阈值**：达到何值判为成功、何值判为失败；可提前设定。
- **预期成本与时间**：人力、时间窗口、其他成本。
- **风险与限制**：本实验不能回答什么、可能干扰因素、归因局限。

---

## Quality bar / Acceptance criteria

- 验证目标与输入的假设**一一对应**；无「先看看反馈」式的开放目标。
- 成功/失败阈值**明确且可观测**；读者能据此判定实验结果。
- 测试方式为**最小可行**；无不必要的全量或高成本设计。
- 指标与假设**对应**；无与假设无关的 vanity 指标主导判定。
- 样本选择与局限已说明；若存在明显偏差风险，已标出。
- 风险与限制节存在；「本实验不能回答什么」已写清。
- 未把「收集定性反馈」当作对关键假设的验证，除非该假设本身就是「用户会表达某类反馈」。

---

## Clarifying questions

在以下情况主动向用户澄清：

- 假设不清晰：要验证的具体陈述是什么？来自 assumption-mapping 吗？
- 资源不明：有多少时间、可触达多少用户、能否做小流量或假门？
- 成功标准不清：业务上如何定义「验证通过」？能否接受「不通过则放弃或 pivot」？

---

## Common pitfalls

- **无判定标准**：设计成「先上线试试看」「先收集反馈」，没有成功/失败阈值，无法判定假设是否成立。
- **指标与假设错位**：验证「用户会不会用」，却用「点击率」当主指标，忽略「是否完成关键行为」。
- **样本失真**：只在内部或自愿者中测，不说明偏差；或样本量过小且未说明局限。
- **成本过高**：一上来全量或复杂多变量，违背最小验证原则。
- **把收集反馈当验证**：用「用户说喜欢」验证 desirability，未设计可观测行为或结果指标。
- **越界写 PRD**：在实验设计中写完整功能需求；本 skill 只产出验证方案，不替代 prd-writing。

## Good output example

**输入**：来自 assumption-mapping 的假设——「目标管理员在获得已配置的快捷入口后，会使用至少 1 次」（即先验证「给了会不会用」，再在后续阶段验证「会不会主动配置」）。

**良好输出（节选）**：

- **Task framing**：本输出为对单条 desirability 假设的验证方案，不替代 PRD。
- **Evidence**：要验证的假设如上；无历史快捷入口数据；可触达约 200 位目标管理员（某渠道）。
- **要验证的假设**：目标管理员中，在**获得已配置好的**快捷入口后，至少 X% 会在 2 周内使用该入口至少 1 次。
- **验证目标**：回答「用户在获得已配置的快捷入口后是否会真用」而非「用户是否说想要」；与假设严格对应。
- **最小测试方式**：Concierge 式——由运营在 2 周内主动为 30 位自愿管理员各配置 1 个快捷入口，观察其是否在 2 周内使用该入口至少 1 次；不建完整产品，不在此阶段验证「主动配置」。
- **样本选择**：30 人，从 200 人池中自愿招募；局限：自愿者可能更积极，结果可能高估。
- **观测指标**：2 周内「使用已配置快捷入口 ≥1 次」的比例；与假设（给了会不会用）直接对应。
- **成功/失败阈值**：成功 ≥40% 使用至少 1 次；失败 &lt;20%；20%～40% 为不确定、建议补样本或延长时间。
- **预期成本与时间**：2 周；1 人运营 + 少量脚本支持。
- **风险与限制**：本实验**不能回答**「用户是否会主动配置」；仅验证「给了会不会用」。自愿者偏差可能高估；Concierge 与真实产品体验有差。
- **Recommendation**：若达到成功阈值，可再设计验证「主动配置」的实验，或接 prd-writing；若失败，建议重新审视问题或方案。
- **Open questions / Next step**：验证结束后是否需对「主动配置」做单独验证？

---

## Bad output example

**输入**：来自 assumption-mapping 的假设——「目标管理员会**主动配置**并使用快捷入口」。若实验设计为「由运营先替用户配置好，再观察是否使用」，则与假设错位。

**不良输出（节选）**：

- **方案**：先上线快捷入口 MVP，全量推给所有管理员，看 2 周数据。
- **指标**：点击率、DAU、用户反馈问卷。
- **目标**：看看用户喜不喜欢、用不用。

问题在于：没有明确成功/失败阈值，无法判定；「先上线全量」成本高、不是最小验证；若假设是「会主动配置并使用」，实验却只观察「给了会不会用」则验证对象错位；指标「点击率」「DAU」与行为假设错位；「用户喜不喜欢」依赖问卷而非行为，易成虚假验证。此类设计无法可靠支撑「假设成立与否」的决策。

## Related skills / Handoff rules

- **assumption-mapping**：本 skill 的输入通常来自 assumption-mapping；若尚未明确要验证的假设，先做 assumption-mapping。路径：`skills/assumption-mapping/SKILL.md`。边界：assumption-mapping 识别假设与验证优先级，experiment-design 设计具体验证方案。
- **prioritization**：验证完成后，若有多条候选或资源约束，可交 prioritization 做排序；本 skill 不负责多候选取舍。
- **prd-writing**：验证通过后，可进入 prd-writing 撰写完整需求；验证未通过或不确定时，建议先迭代假设或再验证，再写 PRD。路径：`skills/prd-writing/SKILL.md`。
- **roadmap-reality-check**：若验证方案被纳入 roadmap，其时间与依赖可由 roadmap-reality-check 做现实性检查；本 skill 不负责检查整体 roadmap。
