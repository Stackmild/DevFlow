# Roadmap Reality Check

本 skill **检查已形成的**阶段性计划、发布时间或路线图是否**脱离现实**，产出：隐藏依赖、顺序风险、资源/容量错配、应在承诺前完成验证的点、以及调整建议。**不**负责在多个候选之间做优先排序——那是 prioritization 的职责；**不**负责制定或修改优先级，只负责对已有计划做现实性审查。输出中须区分 Evidence（已有计划、已知约束）、Inference（从计划推导出的风险）、Recommendation（具体调整建议）。

---

## Best for

- 已有阶段性计划、季度/版本时间线或 roadmap 草案，需要检查是否可执行
- prioritization 完成后，需在落时间线前做一次现实性检查
- 需要识别「计划里没写但会拖垮交付」的依赖与容量问题

---

## Typical triggers

- 用户要求「检查一下这个 roadmap 是否靠谱」「看看时间线是否现实」
- 主 skill 在组合路径中路由（如「做 roadmap」时：prioritization → roadmap-reality-check）
- 计划已排定，上线或承诺前需做一次 sanity check

---

## Not for / When NOT to use

- 需要**在多个候选之间做取舍或排序**时，用 prioritization；本 skill 不产出优先级，只检查已有计划的现实性。
- 尚无任何计划或时间线、仅有一堆候选时，应先做 prioritization，再视需要做本 skill。
- 需要做 launch/expand/rollback 决策时，用 launch-review；本 skill 不负责发布决策。

---

## Required inputs

- 至少包含：**已形成的阶段性计划、发布时间或路线图**（可为列表、时间线、版本规划等）。若完全没有「要检查的对象」，无法执行本 skill，应先产出或提供计划草案。

---

## Optional inputs

- 已知资源或容量（人力、团队、外部依赖）
- 已知依赖关系或前置条件
- 战略或截止时间约束

---

## Default assumptions when context is missing

- 计划中常存在**未写明的依赖**（技术、人力、上游交付、验证结论）；默认需要显式挖出并评估是否已满足或已排期。
- 时间与容量估算常偏乐观；须检查是否有 buffer、并行假设是否成立、验证是否在承诺前完成。
- 不假定「写进计划的就是可执行的」；须独立判断顺序、依赖、容量是否匹配。

---

## Core workflow

1. **通读计划**：理解当前计划的范围、时间点、里程碑与交付物；标注计划中已写明的依赖与假设。
2. **识别隐藏依赖**：列出「计划未写但实现/上线会依赖」的项（如某接口、某权限、某数据、某验证结论）；每项标出当前状态（已有/未就绪/未排期）。
3. **顺序风险**：检查项与项之间的顺序是否合理（A 依赖 B 则 B 是否先于 A）；是否存在「后项已承诺但前项未定」的倒挂。
4. **资源/容量错配**：计划所需人力或能力与可用资源是否匹配；并行项是否真的可并行；是否有未分配的项或过度承诺的窗口。
5. **验证前置条件**：计划中是否有「先验证再承诺」的项；验证是否已排期且在承诺交付前完成；若未验证就承诺，标出风险。
6. **风险与乐观偏差**：是否存在明显的时间或范围乐观；哪些节点最容易滑点。
7. **不可移动承诺时的取舍**：若存在不可移动的对外 deadline 或承诺，须逼出至少一种取舍——降 scope / 改顺序 / 增资源 / 明确接受延期风险；避免只识别风险而不给出可执行选项。
8. **输出分层**：按 Task framing, Evidence, Assumptions, Inference, Analysis, Recommendation, Open questions / Next step 组织；Analysis 内包含上述各项。

---

## Output contract

输出须采用以下结构，并确保「风险」与「建议」可执行。

| 层 | 内容要求 |
|----|----------|
| **Task framing** | 本输出是计划现实性检查，不是优先级排序；边界为「已有计划是否可执行、哪里可能脱节」。 |
| **Evidence** | 输入中的计划内容、已写明的依赖与约束、已知资源或容量。 |
| **Assumptions** | 检查时采用的假设（如「假设当前人力不变」「假设上游按承诺交付」）。 |
| **Inference** | 从计划推导出的风险或隐含前提（如「A 与 B 并行则需 2 人」）；须显式标注。 |
| **Analysis** | 本 skill 核心产出：见下「必含项」。 |
| **Recommendation** | 明确建议，不超过 3 条（如：建议将 X 前置；建议在承诺前补验证 Y；建议某节点加 buffer）。 |
| **Open questions / Next step** | 尚不确定的依赖或容量、建议下一步（如与某方对齐后再更新计划）。 |

**必含项（Analysis 内）**：

- **隐藏依赖**：计划未写但会影响的依赖；每项标出当前是否就绪、是否已排期。
- **顺序风险**：项与项顺序是否合理；是否存在「后承诺、前未定」的倒挂。
- **资源/容量错配**：所需 vs 可用；并行假设是否成立；未分配或过载的窗口。
- **验证前置条件**：应在承诺前完成的验证是否已排期、是否在承诺前；若未满足，标出风险。
- **乐观偏差**：哪些节点或范围最容易滑点；建议的缓冲或降 scope 点。
- **不可移动承诺时的取舍**（若适用）：若存在不可移动的对外承诺或截止时间，须给出至少一种可执行取舍（降 scope / 改顺序 / 增资源 / 明确接受延期风险），不只列风险。

---

## Quality bar / Acceptance criteria

- 输出中**没有**做多候选排序或改写优先级；仅对已有计划做现实性审查。
- 至少覆盖隐藏依赖、顺序、容量、验证前置中的主要风险；无「无风险」敷衍。
- 建议具体可执行（如「将 X 前置」「在 Q2 前完成 Y 验证」），非泛泛而谈。
- 若存在不可移动的对外承诺或截止时间，输出须包含至少一种可执行取舍（降 scope / 改顺序 / 增资源 / 接受延期），不只列风险。
- 证据与推断已区分；读者能看出哪些来自计划、哪些是推导出的风险。

---

## Clarifying questions

在以下情况主动向用户澄清：

- 计划不完整：当前版本是否已包含全部要交付的项？是否有未写进计划的依赖方？
- 资源不明：可用人力或关键角色是否固定？是否有共享资源冲突？
- 截止时间是否硬性：若滑点，是否有不可移动的对外承诺？若有，检查输出是否已逼出至少一种取舍（降 scope / 改顺序 / 增资源 / 接受延期）。

---

## Common pitfalls

- **越界做排序**：改写优先级或「应该先做哪个」；本 skill 只检查现实性，不替代 prioritization。
- **只复述计划**：不识别隐藏依赖与顺序风险，只把计划再列一遍。
- **建议空泛**：写「注意风险」「留足 buffer」而不指出具体节点或具体动作。
- **忽略验证前置**：计划中有依赖验证结论的项，却未检查验证是否在承诺前完成。
- **假定计划可行**：不独立判断容量与依赖，默认「写上了就能做到」。

## Good output example

**输入**：本季度 roadmap——Q1 完成快捷入口上线、Q2 完成搜索排序优化；快捷入口依赖「用户会用」的验证结论。

**良好输出（节选）**：

- **Task framing**：本输出为对当前季度 roadmap 的现实性检查，不修改优先级。
- **Evidence**：计划为 Q1 快捷入口上线、Q2 搜索排序优化；快捷入口在 PRD 中写明依赖「用户会用」验证通过。
- **Assumptions**：假设当前研发 1 人、验证由产品牵头；上游搜索接口按现有承诺 Q1 中可用。
- **Inference**：若验证在 Q1 中才完成，则快捷入口若按 Q1 末上线，开发窗口约 6～8 周；与 1 人容量对比存在紧张。
- **隐藏依赖**：快捷入口依赖「用户会用」验证结论——当前验证计划 2 周内完成，已排期，但若结果不确定需二次验证则可能挤占 Q1 开发。搜索排序依赖搜索接口稳定——已与技术对齐，标为就绪。
- **顺序风险**：无倒挂；验证先于开发承诺，合理。
- **资源/容量错配**：Q1 仅 1 人，快捷入口开发 + 联调 + 灰度若按 8 周估算，与季度窗口匹配但无 buffer；若验证延后则易滑点。
- **验证前置条件**：验证已排期且在开发承诺前；满足。建议验证结论在「开发启动」前锁定，避免开发到一半因验证不确定而返工。
- **Recommendation**：(1) 将验证结论评审设为开发启动的前置 gate。(2) Q1 末节点预留 1～2 周 buffer 或明确可砍范围。(3) 与搜索接口方确认 Q1 中交付日，避免联调阶段阻塞。
- **Open questions / Next step**：验证若为「不确定」是否已有预案？建议与利益相关方对齐后再冻结时间线。

---

## Bad output example

**输入**：同上。

**不良输出（节选）**：

- **检查结果**：计划合理，建议按计划执行。注意控制风险。

问题在于：未识别隐藏依赖（验证结论、搜索接口）；未检查顺序与容量；未指出具体风险点或建议；等同于未做现实性检查。此类输出无法支撑「计划是否可执行」的决策。

## Related skills / Handoff rules

- **prioritization**：本 skill 的输入通常为 prioritization 产出后形成的计划或时间线；若尚无计划、只有候选列表，应先做 prioritization。路径：`skills/prioritization/SKILL.md`。边界：prioritization 负责「多候选间取舍」；本 skill 负责「已有计划的现实性」，不负责排序。
- **experiment-design / assumption-mapping**：若计划中有依赖验证的项，验证方案由 experiment-design 设计；假设识别由 assumption-mapping。本 skill 只检查「验证是否在承诺前排期与完成」。
- **prd-writing**：计划中的交付物若尚未写 PRD，可交 prd-writing；本 skill 不写 PRD，只检查计划现实性。
- **launch-review**：上线或发布决策用 launch-review；本 skill 不负责 launch/expand/rollback/hold 决策。
